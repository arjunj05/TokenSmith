## Project Overview

This project investigates how to improve the context construction step in a local Retrieval-Augmented Generation (RAG) system called TokenSmith. TokenSmith is a tool that lets students query a database textbook and receive answers generated by a local language model. The standard approach retrieves the top-k most relevant text chunks and passes all of them directly to the model. The problem with this approach is that each chunk is scored independently — the system never considers whether the selected chunks work well together as a set.

The goal of this project is to replace fixed top-k selection with a budget-aware joint selection strategy inspired by buffer pool admission control in database systems. Just as a buffer pool manager must decide which pages deserve scarce fast-memory space, this work treats the LLM context window as a fixed resource and asks which combination of chunks produces the most useful context. Each candidate chunk is scored jointly by its relevance to the query, the amount of new information it contributes relative to already-selected chunks, and its token cost. Selection proceeds greedily under a fixed token budget until no more chunks fit.

## Current Progress

**Chunk selector module.** A new module `src/ranking/chunk_selector.py` has been implemented and integrated into the TokenSmith pipeline. It sits between the cross-encoder reranker and the generation step. The selector takes the reranked candidate chunks and greedily selects the best subset under a configurable token budget. Relevance scores come from the reranker. Redundancy is measured as cosine similarity between chunk embeddings, which are retrieved directly from the FAISS index without any recomputation. Token cost is approximated as character count divided by four.

**Configuration.** Three new fields were added to `RAGConfig` and `config.yaml`: `use_chunk_selector` (on/off toggle, default off so existing behaviour is unchanged), `token_budget` (maximum tokens for the context window, default 2000), and `selector_lambda` (trade-off between relevance and novelty, default 0.5).

**Evaluation infrastructure.** The existing LLM-as-judge metric was refactored to fix a reliability issue. The original implementation downloaded the full textbook PDF and sent it with every API call, which caused consistent request timeouts. The updated version grades answers against the reference answers already present in `benchmarks.yaml`, making evaluation fast and stable.

**Benchmark results.** All three conditions were evaluated across 11 benchmark questions using the LLM-as-judge metric:

| Condition | Avg Score (out of 5) |
|---|---|
| Baseline (top-k, no selector) | 2.36 |
| Budget-aware, relevance-only (λ=1.0) | **2.55** |
| Budget-aware, joint with redundancy (λ=0.5) | 2.18 |

The relevance-only budget selector improved on 5 of 11 questions and regressed on 2, for a net gain of +0.19. The joint selector with λ=0.5 showed mixed results, suggesting the redundancy penalty weight needs tuning.

## Challenges and Observations

**Redundancy penalty sensitivity.** The most significant observation is that the λ=0.5 redundancy penalty hurts on questions where the relevant content is naturally concentrated in a few closely related chunks. For example, the functional dependencies question dropped from 5/5 to 3/5. Because all chunks come from the same textbook domain, cosine similarities between retrieved chunks are uniformly high (around 0.85–0.95). A penalty that treats high similarity as redundancy ends up discarding legitimately complementary chunks. The practical implication is that λ should be kept small — enough to act as a tiebreaker between equally relevant candidates, not large enough to override relevance.

**Evaluation reliability.** Running the LLM-as-judge at scale surfaced a fragility in the original design: sending a 700-page PDF on every API request caused systematic 503 timeouts and 429 quota errors. This was resolved by switching to reference-answer-based grading. The lesson is that for iterative experimentation, the evaluation pipeline needs to be fast and cheap enough to run many times.

**Selector has no effect when chunks already fit the budget.** With the current settings (5 reranked chunks averaging ~400 tokens each, budget of 2000 tokens), the total input is often just under the budget, meaning the selector rarely has to make hard trade-offs. A tighter budget or larger candidate pool would make the selection decisions more meaningful.

---

## Next Steps

The immediate next step is a sweep over the λ parameter — testing values like 0.1, 0.2, and 0.3 — to find the range where the redundancy penalty helps rather than hurts. Given the high inter-chunk similarity observed, a small λ is expected to perform better than the 0.5 default.

A second direction is tightening the token budget so the selector is forced to make real trade-offs. Currently most candidate sets fit within the budget without any cuts. Setting the budget to something like 1200–1500 tokens would require the selector to drop 1–2 chunks, making the comparison with the baseline more meaningful.

Finally, the benchmark suite currently has 11 questions. Expanding this to cover more question types and difficulty levels would make the results more robust and better support conclusions about which question categories benefit most from budget-aware selection.
